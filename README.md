# Distributed Work Unit Distributor & Result Commit System

This project is a sample distributed system inspired by Folding@Home. It is built entirely in Go using standard `net/rpc` and demonstrates a decentralized, fault-tolerant master-worker architecture. 

The system automatically handles node failures, prevents race conditions during result submission, and captures consistent global state without halting the entire system.

## Algorithms Implemented

This project fulfills the requirements of a distributed systems course by implementing three classic algorithms:

1. **Leader Election: The Bully Algorithm**
   * **Where it fits:** If the coordinator (Leader) goes down, the remaining worker nodes automatically detect the failure and elect a new coordinator based on the highest Node ID.
   * **Verification:** Visible in the logs when nodes boot up or when the current leader is terminated. Lower-ID nodes will yield to higher-ID nodes.

2. **Mutual Exclusion: Ricart-Agrawala Algorithm**
   * **Where it fits:** To avoid double-commits and race conditions, only one worker at a time is allowed to submit a completed job to the Leader. Workers broadcast requests with Lamport Timestamps and wait for unanimous permission before entering the Critical Section.
   * **Verification:** Visible in the logs via `>>> ENTERED Critical Section! <<<` and `<<< EXITED Critical Section. Granting waiting nodes... >>>`.

3. **Global Snapshot: Chandy-Lamport Algorithm**
   * **Where it fits:** The system periodically takes a consistent checkpoint of the entire distributed state (pending jobs, in-progress jobs, completed jobs, and individual worker status) using "Marker" messages, without freezing the computation.
   * **Verification:** Node 1 automatically initiates a snapshot every 8 seconds. The collected global state is printed as a formatted table in Node 1's terminal.

## How to Run the System (Localhost)

To simulate the distributed network on a single machine, we will run multiple node instances on different ports.

1. Open **three separate terminal windows** (e.g., PowerShell).
2. Navigate to the root directory of this project in all three windows.
3. Start the nodes in reverse order to watch the Bully Algorithm in action:

**Terminal 3 (Becomes Leader):**
```powershell
go run main.go 3
```

**Terminal 2 (Becomes Worker):**
```powershell
go run main.go 2
```

**Terminal 1 (Becomes Worker & Snapshot Initiator):**
```powershell
go run main.go 1
```

## Verification

Once all three nodes are running, observe the terminal outputs to verify the algorithms:

1. Verify Leader Election (Failover)
Go to the terminal running Node 3 (the current Leader) and press Ctrl + C to kill the process.
Look at the terminals for Node 1 and Node 2. You will see them detect the timeout, initiate a new election, and Node 2 will successfully bully Node 1 to become the new Leader and resume distributing jobs.

2. Verify Mutual Exclusion
Watch the worker terminals. You will see them finish processing jobs and explicitly request the Critical Section.
Notice that a worker will never submit a job while another worker is currently inside the Critical Section. They will queue up and grant deferred permissions politely.

3. Verify Global Snapshot
Every 8 seconds, Node 1 will print a === GLOBAL SNAPSHOT === table.
Consistency Check: Add up the Leader's "Pending", "InProgress", and "Completed" job counts. The total will always equal the exact number of jobs generated by the system, proving the snapshot captured a mathematically consistent state across the network, even while jobs were in transit.
